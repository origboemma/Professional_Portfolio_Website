# 1. Purpose

This document defines the standards for creating, reviewing, and approving prototypes for the Professional Portfolio Website.

Prototypes transform design documentation into visual representations that can be evaluated before frontend implementation begins.

The objective is to validate usability, layout, navigation, accessibility, and consistency before any production code is written.

---

# 2. Prototype Objectives

The prototype process aims to:

- Validate page layouts.
- Confirm navigation flows.
- Verify component placement.
- Evaluate responsiveness.
- Improve usability.
- Reduce frontend redesign.
- Increase implementation confidence.

---

# 3. Prototype Levels

The project adopts a three-stage prototype process.

## Level 1 — Low-Fidelity

Purpose:

Validate structure.

Characteristics:

- Basic boxes
- Placeholder content
- No colours
- No branding
- Focus on layout

---

## Level 2 — Mid-Fidelity

Purpose:

Validate interface organisation.

Characteristics:

- Real headings
- Representative content
- Buttons
- Cards
- Typography hierarchy
- Basic spacing

Brand colours remain limited.

---

## Level 3 — High-Fidelity

Purpose:

Represent the final product before development.

Characteristics:

- Complete colour system
- Typography
- Icons
- Images
- Responsive layouts
- Component library
- Design system
- Interactive behaviours

High-fidelity prototypes should closely resemble the final website.

---

# 4. Prototype Scope

Prototypes should be prepared for:

- Homepage
- About Page
- Projects Overview
- Individual Project Page
- Skills Page
- Certifications Page
- Experience Page
- Contact Page
- Navigation
- Footer

---

# 5. Prototype Standards

Each prototype should demonstrate:

- Consistent branding.
- Responsive layouts.
- Correct typography hierarchy.
- Proper spacing.
- Reusable components.
- Clear navigation.
- Accessible interactions.

---

# 6. Annotation Standards

Each prototype should include notes explaining:

- Component purpose.
- Navigation behaviour.
- Responsive changes.
- User interactions.
- Accessibility considerations.
- Implementation notes where necessary.

Annotations should help developers understand design intent.

---

# 7. Prototype Review Criteria

Each prototype should answer the following questions:

- Is the layout intuitive?
- Is navigation clear?
- Are calls-to-action prominent?
- Is information easy to scan?
- Does the design support accessibility?
- Does it align with the Brand Strategy?
- Does it follow the Design System?

---

# 8. Approval Process

Each prototype should pass the following stages:

1. Internal Review
2. Design Review
3. Revision (if required)
4. Final Approval
5. Frontend Implementation

No prototype should move into development without approval.

---

# 9. Handoff to Frontend Development

Approved prototypes should provide developers with:

- Layout guidance.
- Component references.
- Typography rules.
- Colour usage.
- Interaction behaviour.
- Responsive expectations.
- Accessibility requirements.

The prototype serves as the primary implementation reference.

The Prototype Guidelines establish a structured approach for validating the Professional Portfolio Website before development begins.
