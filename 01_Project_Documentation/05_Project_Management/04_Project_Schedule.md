# 1. Purpose

This document defines the project schedule for the Professional Portfolio Website.

The schedule establishes the sequence of project phases, milestones, dependencies, review gates, and progress monitoring activities. It provides a structured roadmap for delivering the project in an organised and controlled manner.

---

# 2. Scheduling Objectives

The project schedule aims to:

- Define the order of project activities.
- Establish measurable milestones.
- Monitor project progress.
- Coordinate deliverables.
- Manage dependencies.
- Reduce scheduling conflicts.
- Support timely completion.

---

# 3. Scheduling Principles

The project schedule follows these principles:

- Documentation before implementation.
- Phase-by-phase execution.
- Milestone-driven progress.
- Quality before speed.
- Review before approval.
- Continuous monitoring.

No implementation activity should begin until the relevant planning and design documentation has been approved.

# 6. Phase Dependencies

Each phase depends on the successful completion of previous phases.

Examples include:

- Brand Strategy depends on Project Initiation.
- Information Architecture depends on Brand Strategy.
- UI/UX Design depends on Information Architecture.
- Software Architecture depends on UI/UX Design and Project Management.
- Frontend Development depends on Software Architecture.
- Testing depends on completed development.
- Deployment depends on successful testing.

Dependencies help ensure project stability and reduce rework.

---

# 7. Progress Monitoring

Project progress will be monitored through:

- Phase completion.
- Milestone achievement.
- Document completion.
- Git commit history.
- GitHub repository updates.
- Review approvals.

Progress should be reviewed at the completion of every major phase.

# 8. Schedule Review

The project schedule should be reviewed whenever:

- A phase is completed.
- Scope changes occur.
- Major risks materialise.
- Milestones require adjustment.
- Significant delays are identified.

Schedule updates should be documented through the Change Management process.
