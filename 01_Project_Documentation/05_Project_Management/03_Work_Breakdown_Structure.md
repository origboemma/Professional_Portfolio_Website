# 1. Purpose

This document defines the Work Breakdown Structure (WBS) for the Professional Portfolio Website.

The WBS decomposes the project into manageable work packages, phases, deliverables, and activities. It provides a structured framework for planning, execution, monitoring, and project control.

---

# 2. Objectives

The Work Breakdown Structure aims to:

- Define the complete project scope.
- Organise work into manageable components.
- Clarify deliverables.
- Support scheduling.
- Improve project monitoring.
- Facilitate resource planning.
- Reduce scope ambiguity.

Every project activity should belong to a defined work package.

---

# 3. WBS Principles

The WBS follows these principles:

- Deliverable-oriented.
- Hierarchical.
- Complete.
- Non-overlapping.
- Traceable.
- Easy to maintain.

The WBS represents **what** will be delivered rather than **how** the work will be performed.

# 5. Work Packages

Each Level 1 deliverable is divided into smaller work packages.

Every work package should:

- Have a defined objective.
- Produce measurable outputs.
- Have clear completion criteria.
- Be independently reviewable.

Work packages form the basis for scheduling and progress tracking.

---

# 6. Deliverable Hierarchy

The project follows the hierarchy below:

Project

↓

Phase

↓

Deliverable

↓

Work Package

↓

Activity

↓

Task

↓

Output

This hierarchy ensures that every task contributes directly to a project deliverable.

---

# 7. Work Package Identification

Each work package should be uniquely identified using a numbering convention.

Example:

| WBS Code | Description |
|----------|-------------|
| 1.0 | Project Initiation |
| 1.1 | Project Charter |
| 1.2 | Scope Definition |
| 2.0 | Brand Strategy |
| 4.3 | Layout System |
| 6.2 | Component Architecture |
| 7.5 | Homepage Development |
| 8.3 | Accessibility Testing |

Unique identifiers improve traceability throughout the project.

---

# 8. Work Package Dependencies

Some work packages depend on the completion of others.

Examples include:

- UI/UX Design depends on Information Architecture.
- Software Architecture depends on UI/UX approval.
- Development depends on Software Architecture.
- Testing depends on completed implementation.
- Deployment depends on successful testing.

Dependencies should be considered during scheduling.

---

# 9. WBS Maintenance

The Work Breakdown Structure should be updated whenever:

- Scope changes are approved.
- New deliverables are introduced.
- Work packages are added or removed.
- Major milestones are revised.

Changes should be documented through the Change Management process.
