# 1. Purpose

This document defines the Change Management process for the Professional Portfolio Website.

It establishes how proposed changes are identified, evaluated, approved, implemented, verified, and documented throughout the software development lifecycle.

A structured change management process helps maintain project stability, protect approved deliverables, and ensure that every significant modification is traceable.

---

# 2. Objectives

The Change Management process aims to:

- Control changes to project scope.
- Protect approved project deliverables.
- Reduce unnecessary rework.
- Ensure proper documentation.
- Maintain project quality.
- Improve decision-making.
- Provide complete traceability for project changes.

---

# 3. Change Management Principles

Project changes follow these principles:

- Every significant change must be documented.
- Every change must be evaluated before implementation.
- Major changes require approval.
- Documentation must remain consistent with implementation.
- Approved changes must be traceable.
- Changes should minimise disruption to the project.

---

# 4. Scope of Change Management

The Change Management process applies to:

- Project scope.
- Business requirements.
- Technical requirements.
- Software architecture.
- UI/UX design.
- Documentation.
- Source code.
- Testing strategy.
- Deployment process.
- Project schedule.

Minor editorial corrections may be implemented without formal approval, provided they do not affect project intent or deliverables.

---

# 5. Change Request Process

Every significant change follows the process below:

1. Submit Change Request.
2. Record the request.
3. Assess the impact.
4. Evaluate alternatives.
5. Approve or reject the request.
6. Plan implementation.
7. Implement the change.
8. Verify the outcome.
9. Update documentation.
10. Close the change request.

No major change should bypass this process.

# 7. Impact Assessment

Every proposed change should be evaluated for its impact on:

- Project scope.
- Schedule.
- Cost (where applicable).
- Technical complexity.
- Quality.
- Risks.
- Documentation.
- Dependencies.

Impact assessments should be evidence-based and documented.

---

# 8. Change Approval

Changes should be approved according to their significance.

| Change Type | Approval Required |
|-------------|-------------------|
| Minor editorial update | Project Manager |
| Documentation improvement | Project Manager |
| UI refinement | Project Manager + UI/UX Designer |
| Architecture modification | Project Owner + Software Engineer |
| Scope expansion | Project Owner |
| Technology replacement | Project Owner + Software Engineer |

No implementation should begin until the required approval has been obtained.

---

# 10. Change Verification

After implementation, each approved change should be verified to ensure that:

- The intended objective has been achieved.
- No unintended effects have been introduced.
- Documentation reflects the implemented change.
- Quality standards remain satisfied.

Verification should be completed before the change is formally closed.

---

# 11. Change Communication

Approved changes should be communicated through:

- Updated documentation.
- Git commit messages.
- GitHub repository history.
- Review records.
- Release notes (where applicable).

Communication should ensure that all relevant stakeholders are aware of significant project changes.

The Change Management document establishes a structured process for managing changes throughout the Professional Portfolio Website project.
