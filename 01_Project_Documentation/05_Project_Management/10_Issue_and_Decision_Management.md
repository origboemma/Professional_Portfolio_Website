# 1. Purpose

This document defines the process for managing project issues and documenting significant project decisions throughout the Professional Portfolio Website.

It establishes a structured approach for recording, analysing, resolving, and tracking issues while maintaining a complete history of important business, design, architectural, and technical decisions.

---

# 2. Objectives

The objectives of Issue and Decision Management are to:

- Record project issues consistently.
- Ensure timely issue resolution.
- Maintain accountability.
- Preserve decision history.
- Improve project transparency.
- Support future maintenance.
- Reduce repeated mistakes.

---

# 3. Guiding Principles

Issue and Decision Management follows these principles:

- Transparency
- Traceability
- Accountability
- Timely resolution
- Evidence-based decisions
- Continuous improvement
- Complete documentation

Every significant issue and decision should be documented.

---

# 4. Issue Management Process

Project issues should follow this workflow:

1. Identify the issue.
2. Record the issue.
3. Assess priority.
4. Assign ownership.
5. Investigate root cause.
6. Implement a resolution.
7. Verify the solution.
8. Close the issue.
9. Record lessons learned.

---

# 5. Issue Classification

Issues may be classified into the following categories:

## Technical Issues

Examples:

- Build failures
- Component defects
- Dependency conflicts
- Performance bottlenecks

---

## Design Issues

Examples:

- Navigation inconsistencies
- Layout problems
- Responsive design defects
- Accessibility concerns

---

## Documentation Issues

Examples:

- Missing documentation
- Inconsistent terminology
- Outdated specifications
- Incorrect diagrams

---

## Process Issues

Examples:

- Delayed reviews
- Incomplete approvals
- Missed milestones
- Communication breakdowns

---

## Deployment Issues

Examples:

- GitHub Pages configuration
- Build pipeline failures
- Deployment errors
- Environment configuration problems

---

# 6. Issue Prioritisation

Issues should be prioritised according to their impact.

| Priority | Description |
|----------|-------------|
| Critical | Prevents project progress or release. |
| High | Major impact requiring prompt resolution. |
| Medium | Noticeable impact but work may continue. |
| Low | Minor issue with limited effect. |

Priority should be reviewed whenever new information becomes available.

# 8. Decision Management

Significant project decisions should be formally documented.

Examples include:

- Architecture selection.
- Technology selection.
- Framework adoption.
- Design direction.
- Folder structure.
- Deployment strategy.
- Coding standards.

Decision records preserve project rationale and improve maintainability.

---

# 9. Decision-Making Process

The following workflow applies to significant project decisions:

1. Identify the decision.
2. Gather relevant information.
3. Evaluate alternatives.
4. Assess impacts.
5. Select the preferred option.
6. Obtain approval.
7. Document the decision.
8. Communicate the outcome.

# 11. Lessons Learned

Every resolved issue and major decision should contribute to the project's knowledge base.

Lessons learned should include:

- What happened.
- Why it happened.
- How it was resolved.
- Recommendations for future work.

Lessons learned should be reviewed during future project planning.
